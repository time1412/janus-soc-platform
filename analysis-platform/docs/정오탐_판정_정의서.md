# 정·오탐(True/False Positive) 판정 정의서
### 분석플랫폼 → 소통플랫폼 이벤트 핸드오프 LLM 판정

| 항목 | 내용 |
|---|---|
| 문서 목적 | 분석플랫폼이 LLM으로 보안 경보의 정탐/오탐을 판정하는 기준과 방식을 정의 |
| 적용 위치 | 분석플랫폼 backend/triage_service.py, /api/triage |
| 판정 시점 | 탐지 경보를 소통플랫폼으로 전달하기 전(1차 판정), 그리고 분석가의 수동 조회 시 |
| 적용 모델 | OpenRouter 경유 gemini-2.5-flash (config.GEMINI_MODEL), temperature 0 |
| 조회 화면 | 분석플랫폼 좌측 "정·오탐" 탭 |
| 작성일 | 2026-06-12 |

---

## 1. 개요

IDS/WAF는 시그니처가 매칭되면 경보를 올리지만, 그중 상당수는 **정상 트래픽이 시그니처에 우연히 걸린 오탐**이다. 본 판정은 LLM이 경보의 **실제 페이로드(URI/본문)를 분석**해 정탐(실제 공격)/오탐(false positive)을 구분하고, **정탐만 소통플랫폼으로 전달**하기 위한 것이다.

- **정탐(true_positive)**: 실제 공격 의도가 페이로드에 드러나는 경보
- **오탐(false_positive)**: 시그니처는 매칭됐으나 정상으로 판단되는 경보

---

## 2. 판정 파이프라인

```
Splunk 탐지(IDS/WAF 경보)
   └─ 분석플랫폼: 경보 페이로드 추출(URI/본문)
        └─ URL 디코딩(%27→' 등)
             └─ LLM 정·오탐 판정 (배치 + 중복제거)
                  ├─ 정탐 → 소통플랫폼 /api/events/ingest 로 자동 전달(티켓화)
                  │         └─ 전송한 건은 캐시 기록 → 재전송 안 함
                  └─ 오탐 → 전달 제외(분석플랫폼 기록만)
```

- 판정은 **분석플랫폼에서 수행**하고, 결과는 분석플랫폼 "정·오탐" 탭에서 조회한다.
- **정탐으로 판정된 건만** 소통플랫폼으로 넘어가 인간 최종 검토(티켓)를 받는다.
- 전달은 판정과 동일한 **백그라운드 주기(60초)**에서 자동 실행된다(아래 §6 전달 정책).

---

## 3. 판정 기준 (Criteria)

> 아래는 `triage_service.py`의 판정 기준 프롬프트(_CRITERIA) 전문이다.

```
[판별 기준]
- 정탐(true_positive): 페이로드/URI에 실제 공격 의도가 명확히 드러남.
  · SQL Injection: ' OR 1=1, UNION SELECT, CTXSYS.DRITHSX, 주석(-- #) 등 SQL 구문
  · XSS: <script>, onerror=, <img src=x>, document.cookie 탈취 등
  · Path Traversal: ../../../etc/passwd, ..%2f 등 상위 경로 탐색
  · Command Injection: ;ls, |cat, $(...), &&whoami 등 OS 명령 결합
  · Brute Force: 동일 출발지에서 인증 엔드포인트(/login 등)로 짧은 간격 반복 시도
- 오탐(false_positive): 시그니처는 매칭됐으나 정상 트래픽일 가능성이 높음.
  · 정상 검색어/파라미터 (예: keyword=bts, keyword=콘서트)
  · 세션 식별자 (예: ;jsessionid=ABC123 — 'id'가 명령어가 아님)
  · 정상적인 파일 경로/확장자, 정상 로그인 1~2회
  · 내부 점검/인증된 스캐너로 보이는 트래픽

[판별 원칙]
1. 반드시 페이로드(URI/본문)의 '실제 내용'을 근거로 판단한다. 시그니처 이름만 믿지 않는다.
2. URL 인코딩은 디코딩된 값으로 해석한다 (%27 → ' , %3C → < ).
3. 애매하면 confidence를 낮추고, 보안상 안전한 쪽(정탐 의심)으로 기운다.
```

**핵심 원칙 요약**
- **시그니처 이름이 아닌 실제 페이로드로 판단** — 같은 "Command Injection" 시그니처여도 `;jsessionid=`(세션ID)는 오탐, `;cat /etc/passwd`는 정탐.
- **디코딩 후 해석** — `%27`(') 등 URL 인코딩을 풀어 공격 구문을 본다.
- **안전 편향(fail-safe)** — 애매하면 신뢰도를 낮추고 정탐 의심으로 기울여, 실제 공격을 놓치지 않는다.

## 4. 판정 방식 (Method)

1. **입력 구성**: 각 경보를 `시그니처 / 출발지·목적지 IP / 탐지원 / 위험도 / URI(디코딩) / 본문(디코딩) / HTTP 응답코드` 텍스트로 변환. **POST 공격은 payload가 URI가 아닌 본문에 있으므로**, ModSecurity 감사 로그의 요청 본문(C 섹션)을 추출해 함께 입력한다. (이 본문이 없으면 `/qna/searchQnaList.do` 같은 정상 경로만 보여 SQLi를 오탐할 수 있음)
2. **상관(correlation) 병합 — "한 공격(버스트) = 한 인시던트"**: 키는 **`(정규화된 공격유형 + 출발지 IP + 시간 윈도우)`** 다. 시그니처 표기("XSS" / "XSS (Snort)" / "[YANUS CUSTOM] [XSS] …" / "ET WEB_SERVER Script tag…")는 모두 `XSS` 한 유형으로 정규화된다.
   - **시간 윈도우(기본 30분, `TRIAGE_MERGE_WINDOW_MIN`)**: 같은 유형·IP라도 윈도우가 다르면 **새 인시던트**다. 멀티센서·GET/POST 등 동일 사건의 분산 탐지(수 초 내)는 같은 윈도우라 묶이고, **30분이 지난 새 공격은 새 티켓으로 전달**된다. (윈도우가 없으면 유형+IP가 영구 병합되어 신규 공격이 기존 인시던트에 흡수·누락되는 문제가 있었음 → 시간 윈도우로 해결)
   - **왜 payload 지문이 아니라 유형+IP+시간인가**: Snort/IDS 로그는 룰 이름만 남기고 **실제 요청 payload를 담지 않는다**. payload 지문으로는 WAF↔Snort를 이어 붙일 수 없어, 출발지 IP + 공격유형 + 시간 윈도우로 상관한다.
   - **대표 경보 판정**: 그룹 내에서 **payload 증거가 가장 풍부한 경보(주로 WAF Web)**를 골라 그 1건만 LLM 판정 → 판정을 그룹 전체에 적용. 룰명만 있는 Snort 건이 대표로 뽑혀 근거 부족으로 오판하는 것을 막는다.
   - **트레이드오프**: 같은 윈도우·같은 유형의 **다른 payload**(예: 서로 다른 XSS 2종)는 한 인시던트로 합쳐진다(윈도우를 짧게 하면 완화). 개별 payload는 그룹 내 indicators/시그니처 목록으로 보존한다.
3. **배치 호출**: 신규 인시던트의 대표 경보를 최대 12건씩 묶어 한 번의 LLM 호출로 판정(JSON 배열 응답). 시스템 프롬프트 비용 분산.
4. **결정성**: `temperature = 0` 으로 동일 입력에 동일 판정(일관성 확보).
5. **구조화 출력 파싱**: 모델이 JSON으로만 답하도록 강제하고, 마크다운 펜스 제거 후 파싱. 파싱 실패 시 "판별불가"로 처리하고 **정탐 쪽(수동 검토 필요)으로 안전 처리**.
6. **판정 매핑**: 인시던트(그룹) 판정을 그룹에 속한 모든 원본 경보에 매핑하고, 결과는 그룹당 1행으로 반환(병합 건수·소스 목록 포함).
7. **자동 판정(중요)**: 분석가가 수동 실행하지 않는다. 경보가 분석플랫폼에 들어오면 **백그라운드 작업이 주기적으로(기본 60초) 신규 인시던트를 자동 판정**한다. "정·오탐" 탭은 그 결과를 보여줄 뿐이다.
8. **캐시**: 판정 결과를 `(정규화된 공격유형 + 출발지 IP + 시간 윈도우)` 키로 캐시(`storage/triage_cache.json`)해 **동일 인시던트 재판정을 막는다** → 신규 인시던트(새 윈도우 포함)만 LLM 호출. 전달 중복 방지(`storage/forwarded_keys.json`)도 같은 키를 쓰므로, **새 윈도우의 공격은 자동으로 재전달**된다.

---

## 5. 출력 스키마

각 경보 판정 결과(JSON):

| 필드 | 의미 |
|---|---|
| verdict | "정탐" 또는 "오탐" |
| confidence | 0~100 신뢰도(정수) |
| confidence_reason | **그 신뢰도 점수가 나온 이유(한 문장)** |
| attack_type | 확정 공격 유형(오탐이면 "해당없음") |
| reasoning | 판단 근거 1~2문장(한국어) |
| indicators | 근거가 된 페이로드 조각들(배열) |
| recommended_action | 권고 조치(한 문장) |
| is_true_positive | verdict가 "정탐"이면 true (전달 판단용 파생값) |

## 6. 신뢰도(confidence) 책정 기준

신뢰도는 LLM이 **아래 고정 루브릭에 따라** 매기고, 각 건마다 그 점수가 나온 이유를 `confidence_reason`에 한 문장으로 남긴다. (단순 임의값이 아니라 기준에 근거한 자기평가)

| 점수 | 의미 |
|---|---|
| 90~100 | 페이로드에 공격 구문이 명확히 존재 — 이론의 여지 거의 없음 |
| 70~89 | 공격 정황이 강하나 일부 모호하거나 컨텍스트 부족 |
| 50~69 | 정탐/오탐 판단이 애매하여 추가 확인 필요 |
| 0~49 | 근거가 약하거나 정보 부족 |

- 예: "디코딩된 URI에 UNION SELECT 구문이 그대로 존재 → 100에 근접" / "정상 페이지네이션 파라미터만 있어 공격 근거 없음".
- `temperature=0`이라 동일 입력엔 동일 점수가 재현된다.
- **주의**: 신뢰도는 실제 정확도로 통계 보정된 값이 아니라 **기준 기반 모델 자기평가**다. 따라서 자동 차단의 하드 임계값으로 쓰지 않고, **낮은 신뢰도는 소통플랫폼에서 우선 검토** 신호로 사용한다.

### 전달 정책
- **전달 기준**: `is_true_positive == true` 인 경보만 소통플랫폼으로 전달한다.
- **자동 전달**: 백그라운드 작업(60초)이 자동 판정 직후 정탐을 소통플랫폼 `POST /api/events/ingest`로 보낸다(`forwarder_service.py`). 분석가가 누르지 않아도 적재된다.
- **중복 전송 방지**: 전송한 경보 키`(시그니처+출발지IP+디코딩URI)`를 `storage/forwarded_keys.json`에 기록해 **같은 정탐을 매 주기 재전송하지 않는다**. 동일 공격 반복은 1건으로 묶고 `dup_count`로 합산해 보낸다. 전송 실패 시 키를 기록하지 않아 다음 주기에 자동 재시도한다.
- **수동 전달(멱등)**: 필요 시 `POST /api/forward`로 즉시 전달할 수 있으며, 이미 보낸 건은 다시 보내지 않는다.
- **안전 편향**: 판별불가/애매 건은 오탐으로 버리지 않고 정탐(수동 검토)로 보낸다 → 미탐(false negative) 방지.
- **인간 최종 판단**: LLM 판정은 1차 자동 분류이며, 소통플랫폼에서 분석가가 승인/반려로 최종 확정한다.
- **수신 결과**: 소통플랫폼은 수신 즉시 티켓번호(`INC-YYYYMMDD-####`)·우선순위(P1~P4, severity 기반)·SLA 기한을 부여하고 상태 "대기"로 적재, 실시간으로 관제팀에 알림(WebSocket)한다.

## 7. 토큰·비용 최적화

- **중복 제거 + 캐시**로 동일/반복 경보는 재판정하지 않음(예: 브루트포스 200건 → 1건만 LLM 호출).
- **배치 호출**로 시스템 프롬프트 1회를 여러 건에 분산.
- 자동 판정은 **신규 고유 경보만** LLM을 호출하므로, 새 공격이 없으면 추가 비용이 발생하지 않는다.

---

## 8. 조회 위치 (분석플랫폼)

- 분석플랫폼 좌측 패널 **"정·오탐" 탭** → 진입 시 **자동 판정 결과가 표시**된다(수동 실행 불필요, 주기적 자동 갱신).
- 표시: 정탐/오탐 건수 요약, 정탐/오탐 필터, 경보별 **판정·신뢰도·신뢰도 근거·판단 근거·핵심 지표(indicators)**.
- API: `GET /api/triage`(자동 판정 결과 조회). 백그라운드 작업이 경보 유입 시 자동 판정·캐시.

## 9. 한계 및 운영 고려사항

- **브루트포스/DoS 주의**: 이들은 '반복 횟수'가 핵심 증거다. 중복 제거로 1건만 보면 1회 로그인처럼 보여 오탐 판정될 수 있다 → **합쳐진 건수(dup_count)를 함께 입력**하는 것을 권장.
- **환각 잔존**: 프롬프트로 억제하나 0은 아님. 안전 편향 + 인간 최종 검토로 보완.
- **외부망 의존**: OpenRouter(openrouter.ai) 외부 호출 필요 → 폐쇄망은 방화벽 허용 또는 내부 LLM. (분석플랫폼은 방화벽 룰로 외부 허용)
- **모델 의존**: 모델 교체 시 고정 샘플셋으로 정확도 재검증(정탐4+오탐3 등) 필요.
- **검증 책임**: 자동 판정은 참고용, 전달·차단의 최종 책임은 분석가에게 있다.
