### SOC 분석플랫폼 — AI 보고서 생성 · 정·오탐 판정

| 항목 | 내용 |
|---|---|
| 문서 목적 | 분석플랫폼의 AI(보고서 생성·정오탐 판정)에 사용되는 프롬프트의 구조·설계 원칙·사용 방식·개선 내역을 정의 |

## 1. 개요

분석플랫폼은 Splunk에서 수집·탐지한 보안 이벤트를 LLM(대규모 언어모델)에게 분석시켜 보고서(텍스트/PDF)로 생성하고, 경보가 들어오면 정·오탐을 자동 판정한다. 이때 모델의 출력 품질·일관성·신뢰성은 전적으로 프롬프트 설계에 좌우된다. 본 정의서는 현재 운영 중인 프롬프트 4종을 기준으로 그 설계 의도와 사용 규격을 명문화한다.

프롬프트 4종:

| 식별자 | 산출물 | 출력 형식 | 위치 |
|---|---|---|---|
| _SYSTEM_PROMPT | 공격 분석 보고서(본 보고서) | 마크다운 | gemini_service.py |
| _TRENDS_SYSTEM | 위협 트렌드 해석 | 마크다운 | insights_service.py |
| _CHAT_SYSTEM_PROMPT | (참고) 로그 분석 챗봇 | 마크다운 | gemini_service.py |
| _BATCH_SYSTEM_PROMPT | 정·오탐 자동 판정 | JSON 배열 | triage_service.py |

> 앞의 3종(보고서·트렌드·챗봇)은 **사람이 읽을 마크다운**을 출력하고, 4번째(정·오탐 판정)는 **기계가 파싱할 JSON**을 출력한다는 점에서 출력 계약이 근본적으로 다르다.

## 2. 프롬프트 아키텍처

모든 호출은 2-메시지 구조(시스템 + 사용자)를 따른다.

```
[시스템 메시지]  = 역할(페르소나) + 작업 지시 + 출력 형식 + 제약(환각 방지)   ← 고정 템플릿
[사용자 메시지]  = 분석 대상 데이터(이벤트 JSON / 집계 통계) + 보강 컨텍스트(TI)  ← 런타임 주입
```

보고서 생성 데이터 흐름:

```
Splunk 탐지 로그
   └─(SPL 집계/조회)→ 이벤트·통계
        └─ 위협 인텔리전스(TI) 보강 [AbuseIPDB/OTX]  → context 문자열
             └─ 프롬프트 조립(시스템+사용자) → LLM 호출(OpenRouter/Gemini)
                  └─ 마크다운 응답 텍스트
                       ├─ 웹: react-markdown 렌더
                       └─ PDF: report_generator.py(마크다운 파싱→스타일 렌더)
```

핵심 설계: 출력 계약(Output Contract)이 곧 소비처 사양과 연결된다.
- **보고서 계열(마크다운)**: 사람이 읽음 → 소비처(웹 react-markdown / PDF report_generator)가 렌더링.
- **정·오탐 판정(JSON)**: 기계가 파싱 → 후단(소통플랫폼 자동 전달 / 필터 / 캐시)이 결정적으로 소비.

정·오탐 판정 데이터 흐름:

```
Splunk 경보
   └─ URL 디코딩 + 중복제거 + 배치 조립(번호 #N)
        └─ LLM 호출(JSON 강제) → 펜스 제거 후 파싱
             ├─ 정탐 → 소통플랫폼 자동 전달(중복 전송 방지 캐시)
             └─ 오탐 → 전달 제외(분석플랫폼 기록만)
```

## 3. 프롬프트 카탈로그

### 3.1 공격 분석 보고서 — _SYSTEM_PROMPT

- 용도: /api/analyze 및 Splunk 웹훅(/webhook/splunk) 수신 시, 이벤트를 분석해 정식 공격 분석 보고서를 생성. PDF로 출력됨.
- 호출: GeminiService.analyze_incident(events, context)
- 시스템 프롬프트 전문:

```
당신은 SOC(보안관제센터)의 시니어 침해대응 분석가입니다.
주어진 보안 이벤트 로그를 분석하여 다음 항목을 한국어로 작성하세요.

1. 사건 요약 (무엇이 탐지되었는가)
2. 공격 유형 분류 (MITRE ATT&CK 전술/기법 매핑 포함)
3. 위험도 평가 (낮음/중간/높음/심각 중 하나와 근거)
4. 영향받은 자산 및 IP
5. 즉시 조치사항 (우선순위 순)
6. 재발 방지 권고사항

추측은 명확히 '추정'으로 표기하고, 로그에 근거한 사실과 구분하세요.
```

- 사용자 메시지 조립(_build_prompt):

```
[추가 컨텍스트]
{TI 보강 텍스트}            ← 있을 때만

[분석 대상 보안 이벤트 ({N}건)]
{events_json}

위 이벤트를 분석하여 공격 분석 보고서를 작성하세요.
```

- 출력 계약: 6개 고정 섹션, 한국어, 마크다운(헤더/불릿). 위험도는 낮음/중간/높음/심각 중 1개.
- 설계 의도:
  - 고정 6항목 → 보고서 구조 표준화(분석가/PDF 렌더가 예측 가능).
  - MITRE ATT&CK 매핑 강제 → 위협 분류 체계 표준 준수.
  - 위험도 라벨 한정(enum) → 후단(PDF 위험도 배지)에서 색상 매핑 가능.
  - "추정" 명시 규칙 → 환각(hallucination) 억제 / 사실·추측 분리.

### 3.2 위협 트렌드 해석 — _TRENDS_SYSTEM

- 용도: /api/insights/trends?days=N. 기간별 공격 집계(시그니처·심각도·일별·총계)를 받아 트렌드를 해석.
- 시스템 프롬프트 전문:

```
당신은 SOC 위협 인텔리전스 분석가입니다.
제공된 보안 이벤트 집계 통계를 분석하여 다음을 한국어로 작성하세요.

1. 이번 기간 주요 위협 트렌드 (증가·감소 패턴, 수치 인용)
2. 가장 심각한 위협 유형과 이유
3. 반복 공격 캠페인 징후 (있다면)
4. 분석가가 즉시 확인해야 할 사항

추측은 반드시 '추정'으로 명시하고 간결하게 작성하세요.
```

- 사용자 메시지: [{N}일간 공격 집계]\n{stats JSON} (by_signature/by_severity/daily_trend/total)
- 설계 의도: 개별 이벤트가 아닌 집계 통계를 입력 → 토큰 절약 + "패턴/추세" 관점 유도. "수치 인용" 지시로 근거 기반 서술 강제.

### 3.3 (참고) 로그 분석 챗봇 — _CHAT_SYSTEM_PROMPT

- 용도: /api/chat. 수집된 로그를 컨텍스트로 자연어 질의응답.
- 시스템 프롬프트 전문:

```
당신은 SOC(보안관제센터) 로그 분석을 돕는 챗봇입니다.
제공된 '수집된 보안 로그'를 근거로 사용자의 질문에 한국어로 간결하고 정확하게 답하세요.
- 로그에 근거가 있으면 구체적 수치/IP/시그니처를 인용해 답합니다.
- 로그에 없는 내용은 추측하지 말고 "수집된 로그에는 해당 정보가 없습니다"라고 답합니다.
- 답변은 요점 위주로 짧게. 표가 도움이 되면 간단한 목록을 사용하세요.
```

- 설계 의도: RAG식 grounding — "제공된 로그만 근거", 없으면 명시적 거부 문구. 대화이력(history)·이벤트(최근 24h)를 사용자 메시지에 함께 주입.

### 3.4 정·오탐 자동 판정 — _BATCH_SYSTEM_PROMPT

- 용도: 경보가 분석플랫폼에 들어오면 백그라운드(60초)가 자동으로 정탐/오탐을 판별. 정탐만 소통플랫폼으로 전달. (조회 `GET /api/triage`)
- 호출: TriageService.classify_batch(alerts) — 고유 경보를 최대 12건씩 묶어 배치 판정
- 다른 프롬프트와의 결정적 차이: **출력이 마크다운이 아니라 JSON 배열**이다(기계가 파싱해 전달/필터/캐시에 사용). 따라서 출력 계약이 매우 엄격하다.
- 시스템 프롬프트 구조(_CRITERIA + _CONFIDENCE_RUBRIC + 출력 스키마 조립):

```
당신은 SOC(보안관제센터)의 시니어 침해대응 분석가입니다.
IDS/WAF가 탐지한 보안 경보 '여러 건'을 각각 정탐/오탐으로 판별합니다.

[판별 기준]
- 정탐(true_positive): 페이로드/URI에 실제 공격 의도가 명확히 드러남.
  · SQL Injection / XSS / Path Traversal / Command Injection / Brute Force …
  · 그 외(SSRF·XXE·SSTI·역직렬화 등)도 목록에 없더라도 공격 의도가 보이면 정탐.
- 오탐(false_positive): 시그니처는 매칭됐으나 정상 트래픽(정상 검색어,
  ;jsessionid= 세션ID, 정상 경로, 정상 로그인, 내부 스캐너 등).

[판별 원칙]
1. 시그니처 이름이 아니라 페이로드(URI/본문) '실제 내용'으로 판단.
2. URL 인코딩은 디코딩한 값으로 해석(%27→' , %3C→<).
3. 목록에 없는 공격도 일반 원칙으로 판정(예시는 화이트리스트가 아님).
4. 애매하면 confidence를 낮추고 안전한 쪽(정탐 의심)으로 기운다.

[신뢰도(confidence) 책정 기준]
- 90~100 : 공격 구문이 명확 — 이론의 여지 거의 없음
- 70~89  : 공격 정황 강하나 일부 모호
- 50~69  : 애매하여 추가 확인 필요
- 0~49   : 근거 약함/정보 부족
→ 각 건마다 점수를 매기고 '왜 그 점수인지'를 confidence_reason에 한 문장 기재.

입력된 모든 경보를 판별해 'JSON 배열 하나만' 출력한다. 다른 설명/마크다운 금지.
각 원소는 입력 경보 번호(id)와 1:1 대응해야 한다.
[ {"id":1, "verdict":"정탐|오탐", "confidence":0~100,
   "confidence_reason":"신뢰도 점수의 근거(한 문장)",
   "attack_type":"확정 공격 유형(오탐이면 '해당없음')",
   "reasoning":"판단 근거(1~2문장)",
   "indicators":["근거가 된 페이로드 조각들"],
   "recommended_action":"권고 조치(한 문장)"}, ... ]
```

- 사용자 메시지 조립(_build_batch_prompt): 경보별 블록(시그니처·출발지/목적지 IP·탐지원·위험도·디코딩 URI·디코딩 본문)을 번호(#1, #2 …)로 나열하고 "id를 포함한 JSON 배열로만 출력하세요."로 마무리.
- 출력 계약: JSON 배열, 원소 수 = 입력 경보 수, id로 1:1 매핑. 각 원소는 위 8개 필드.
- 설계 의도:
  - **JSON 강제 + 펜스 제거 후 파싱** → 후단(전달/필터/캐시)이 결정적으로 소비. 파싱 실패 시 "판별불가"로 처리하고 안전하게 정탐(수동 검토)로 폴백.
  - **신뢰도 루브릭 내장(enum식 구간)** → 점수가 임의값이 아니라 기준 기반이 되도록 고정. `temperature=0`과 결합해 동일 입력 동일 점수 재현.
  - **confidence_reason 강제** → "왜 그 신뢰도인지"를 한 문장으로 남겨 설명가능성(explainability) 확보.
  - **배치(여러 건 1회 호출) + id 1:1 매핑** → 시스템 프롬프트 비용을 분산하고 fan-out 매핑을 가능케 함.
  - **목록 외 공격도 판정(화이트리스트 아님)** → 예시에 없는 공격을 오탐으로 흘리지 않게 일반화 지시.
- 판정 기준·전달 정책의 상세는 별도 문서 `정오탐_판정_정의서`를 참조(본 문서는 프롬프트 설계 관점만 다룬다).

## 4. 공통 프롬프트 엔지니어링 원칙

| 원칙 | 적용 방식 | 효과 |
|---|---|---|
| 역할 페르소나 | "시니어 침해대응 분석가 / 위협 인텔 분석가" | 답변의 전문성·관점·톤 고정 |
| 구조화 출력 | 번호 항목·고정 형식 명시 | 일관된 보고서 양식, 후단 렌더 용이 |
| 환각 방지(grounding) | "로그 근거", "추측은 '추정' 표기", "없으면 정보 없음" | 허위 정보 억제, 신뢰성 |
| 언어 고정 | "한국어로 작성" | 출력 언어 일관성 |
| 분류 체계 표준 | MITRE ATT&CK 매핑 요구 | 위협 분류 표준 준수 |
| enum 제약 | 위험도 낮음/중간/높음/심각, 신뢰도 구간(90+/70+/50+/0+) | 후단(배지 색상·우선검토) 매핑 가능 |
| 구조화 출력(JSON) | 정·오탐 판정은 JSON 배열 강제 + id 1:1 매핑 | 기계 파싱·자동 전달/필터/캐시 |
| 설명가능성 | confidence_reason(신뢰도 근거 한 문장) 강제 | 판정 근거 추적·검토 |
| 결정성 | temperature=0 | 동일 입력 동일 출력(재현성) |
| 용도별 분리 | 보고서(상세)·트렌드(집계)·챗봇(QA)·판정(JSON) 프롬프트 분리 | 목적별 최적화 |
| 컨텍스트 보강 | TI(평판·캠페인) 주입 | 분석 깊이 향상 |

TI 보강 컨텍스트 형식 (main.py:_format_ti_context):
```
[위협 인텔리전스 보강]
- {IP}: 위험도 {score}/100, 신고 {n}건, 국가 {국가코드}, ISP {isp}, 캠페인: {이름들}
```

## 5. 데이터 주입 규격 (변수)

| 변수 | 출처 | 형식 | 사용 프롬프트 |
|---|---|---|---|
| events_json | Splunk 조회 결과 | JSON 배열(시그니처·IP·URI·severity 등) | 공격 분석 보고서 |
| context | TI 보강 | 텍스트 블록 | 공격 분석 보고서 |
| stats | Splunk 집계 | total, by_signature, by_severity, daily_trend | 트렌드 해석 |
| history/events | 대화·최근 로그 | 텍스트/JSON | 챗봇 |
| alert 블록 | 경보(중복제거 후 고유 건) | 시그니처·출발지/목적지 IP·탐지원·위험도·디코딩 URI/본문, 번호(#N) | 정·오탐 판정 |

## 6. 개선 내역 및 버전 관리 방침

현재 프롬프트는 코드 상수로 관리되며 공식 버전 이력은 별도로 추적되지 않는다. 아래는 코드에 반영된 설계 결정(개선 포인트)을 정리한 것이다.

설계 결정(왜 현재 형태가 되었나):
1. 보고서 항목을 6개로 고정 — 자유서술 대비 일관성·후단 PDF 렌더 안정성 확보.
2. 위험도 라벨을 enum으로 제약 — PDF 위험도 배지(빨강/주황/초록) 자동 매핑을 위해.
3. "추정" 분리 규칙 도입 — 초기 자유서술이 사실/추측을 섞던 문제(환각) 대응.
4. MITRE ATT&CK 매핑 명시 — 분류 표준화 요구.
5. 용도별 프롬프트 분리(보고서·트렌드·챗봇·판정) — 목적에 맞춘 길이·관점·출력형식 차별화.
6. 보고서 계열 출력은 마크다운으로 통일 — 웹(react-markdown)·PDF(report_generator) 양쪽에서 렌더 가능.
7. 정·오탐 판정 프롬프트 신설(JSON 출력) — 사람이 읽는 보고서와 달리 기계가 소비(자동 전달/필터/캐시)하므로 JSON 배열 + id 1:1 매핑 + 펜스제거 파싱으로 설계.
8. 신뢰도 루브릭 + confidence_reason 도입 — 초기에는 모델이 임의 신뢰도만 내놓아 일관성·설명력이 부족 → 점수 구간을 고정하고 근거 한 문장을 강제.

본 프로젝트에서 확인/조정된 사항:
- AI 출력의 마크다운 특수문자(**, ###)가 화면에 그대로 노출되던 문제 → 소비처(웹 UI)에서 마크다운 렌더링으로 해결. 즉 프롬프트의 출력 형식(마크다운)과 렌더러 사양을 일치시켜야 함을 명문화.
- 경영진 주간 위협 요약 기능은 운영에서 제외됨 → 관련 프롬프트는 본 정의서 범위에서 제외.
- 정·오탐 판정에서 "정의된 몇몇 공격만 본다"는 한계 → 프롬프트에 "목록은 화이트리스트가 아님, 목록 외 공격도 일반 원칙으로 판정" 지시를 추가해 일반화.

권고: 향후 프롬프트 버전 관리 방안
- 각 프롬프트에 버전 태그/주석(예: # v1.2 2026-06-xx 변경: …) 부여
- 변경 시 변경 이력 표(날짜·사유·이전/이후 diff) 작성
- 모델 교체·프롬프트 수정 시 고정 샘플 이벤트셋으로 회귀 비교(출력 품질 A/B)
- 프롬프트를 코드 상수 → 설정 파일/DB로 분리해 재배포 없이 갱신 가능하게 고려

## 7. 출력 처리(후처리) 규격

report_generator.py가 AI 마크다운을 PDF로 렌더링한다:
- **굵게**, ### 헤더, *불릿*, --- 구분선 파싱 → 스타일 적용
- 커버 배너 / 섹션 헤더 / 번호·불릿 목록 / 위험도 배지(낮음·중간·높음·심각 키워드 인식) / 이벤트 테이블
- 따라서 프롬프트의 출력 형식(섹션·위험도 라벨)이 PDF 시각 요소와 직접 연동된다.

## 8. 한계 및 운영 고려사항

- 모델 의존성: 출력 품질은 모델(gemini-2.5-flash)에 의존. 모델 교체 시 프롬프트 재검증 필요(보고서·판정 모두).
- 외부망 의존(중요): 보고서·트렌드·챗봇·정오탐 판정 모두 OpenRouter(openrouter.ai) 외부 호출. 폐쇄망에서는 방화벽 허용 또는 내부 LLM 필요. (분석플랫폼은 방화벽 룰로 외부 허용 예정)
- 토큰 비용: 이벤트 다수를 JSON으로 주입하므로 입력 토큰이 큼. 집계(트렌드)는 통계만 주입해 절감. 정·오탐 판정은 **중복제거 + 배치 + 캐시**로 신규 고유 경보만 호출해 절감.
- JSON 파싱 견고성: 정·오탐 판정은 모델이 가끔 펜스(```)·여분 텍스트를 붙일 수 있어 펜스 제거 후 파싱하고, 실패 시 안전하게 정탐(수동 검토)로 폴백한다.
- 신뢰도의 성격: confidence는 통계 보정값이 아니라 **루브릭 기반 모델 자기평가**다 → 자동 차단의 하드 임계값으로 쓰지 않는다(상세는 정오탐_판정_정의서 §6).
- 검증 책임: AI 산출물은 참고용이며 분석가의 사람 검토를 전제로 한다(특히 위험도·조치 권고·정오탐 최종 확정).
- 환각 잔존: 프롬프트로 억제하나 0은 아님 → "추정" 표기·근거 인용·confidence_reason 규칙을 운영적으로 강제.
